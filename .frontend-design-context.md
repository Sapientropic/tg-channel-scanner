# TGCS Frontend Design Context

## Visual Thesis

TGCS dashboard should feel like a premium retro pixel intelligence console: dark local-machine shell, warm report-paper panels, crisp 1px borders, hard pixel shadows, amber/teal/rust/blue semantic accents, and dense scan-friendly information.

## UX Layout Contract

- Use a three-zone workbench: left navigation/status, center work queue, right operational context.
- Header is brand and local-state only; it must not become a marketing hero.
- Keep the right rail action-led. It should not restate queue, run, or privacy readouts already shown in the header, command strip, or active board.
- Empty state should render as an operational scanning desk, not a blank box.
- Review cards are the primary interaction surface; their action area should feel deliberate and stable.
- Brand mark should be a product icon, not TG initials: Telegram-style signal plane, radar scan, and inbox/review tray are the preferred visual primitives.
- Brand mark click target opens the Git repository by default. Update/pull behavior lives in the explicit Repository panel: check remote state first, then allow confirmed fast-forward pull only when the working tree is clean.
- Motion should feel like a workstation booting and scanning: entrance sequence, radar/scan lines, queue meter movement, and tactile hover states. Always respect reduced-motion.

## Product Boundary

- Dashboard is optional and local-first; CLI and JSON contracts remain primary.
- The UI should show local/privacy state clearly without exposing bot tokens or raw Telegram text.
- Empty state is valid product state and should still feel operational, not unfinished.

## Avoid

- Generic white SaaS cards.
- Decorative gradient blobs, glassmorphism, or one-hue palettes.
- Fake sample intelligence items in the real dashboard.
- Visible setup tutorials inside the primary app surface.
