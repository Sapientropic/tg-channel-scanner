# TGCS Frontend Design Context

## Visual Thesis

TGCS dashboard should feel like a premium retro pixel intelligence console: dark local-machine shell, warm report-paper panels, crisp 1px borders, hard pixel shadows, amber/teal/rust/blue semantic accents, and dense scan-friendly information.

## Cross-Surface Contract

- Dashboard and HTML Report should feel like the same product family, but not the same surface. Dashboard is a dark local-machine triage console; Report is a readable pixel deck for saved review.
- Shared DNA: OKLCH semantic signal colors, hard pixel shadows, crisp square controls, CJK-readable body typography, and action-first information hierarchy.
- Do not make Dashboard depend on Report-only assets, remote fonts, or browser font cache. Report may load its own presentation fonts; Dashboard must remain local-first and offline-friendly.
- Report must not depend on Dashboard runtime state, local-only APIs, or ephemeral UI modes. It should render deterministically from its JSON contract and embedded assets.
- Display typography should be technical sans across Dashboard and Report. Dashboard uses local/system fallback; Report can use its own loaded display font.
- Semantic signal colors must transfer across surfaces: high/apply/signal-positive, medium/inspect/caution, low/skip/archive, danger/destructive. Avoid introducing a new hue meaning for one local component.
- Low-alpha surface washes may use OKLCH `color-mix()` in the primary path, but fallbacks must preserve usable contrast in older or unusual rendering paths.
- Shadows should remain hard, near-black, and slightly cool. They do not need to track every ink token change 1:1, because muddy shadows make signal colors harder to scan.

## UX Layout Contract

- Use a two-zone review workbench by default: left navigation/status and center work queue. Add a right operational context only when that view has concrete actions for it, such as Settings repository controls.
- Header is brand and local-state only; it must not become a marketing hero.
- Keep any right rail action-led. It should not restate queue, run, or privacy readouts already shown in the header, command strip, or active board.
- Empty state should render as an operational scanning desk, not a blank box.
- First Useful Report / setup checklist is allowed only in empty, blocked, or setup-recovery states. Normal review screens should open on the current queue, latest action, or all-clear judgment instead of visible tutorial copy.
- CommandStrip / latest-run summary belongs inside the active board column, not as an app-wide banner. When latest actionable cards exist, the queue itself is the summary; do not render duplicate global status blocks above it.
- Inbox defaults to the Latest action filter: high-priority new/changed cards from the latest run. If that explicit filter is empty, show an inline next step such as switching to All; do not auto-redirect the user's filter selection.
- Review cards are the primary interaction surface; their action area should feel deliberate and stable.
- Review-card diff authoring is a mode change, not another oversized button row. When `Draft diff` is expanded, keep primary review decisions in a compact action cluster and move note writing into a distinct composer with a secondary `Hide diff` control.
- Expanding the diff composer should focus the note textarea and scroll it into view. Do not make users hunt below the fold after clicking `Draft diff`.
- Brand mark should be a product icon, not TG initials: Telegram-style signal plane, radar scan, and inbox/review tray are the preferred visual primitives.
- Brand mark click target opens the Git repository by default. Update/pull behavior lives in the explicit Repository panel: check remote state first, then allow confirmed fast-forward pull only when the working tree is clean.
- Motion should feel like a workstation booting and scanning: entrance sequence, single-pass radar/scan confirmation, queue meter movement on state load, and tactile hover states. Avoid ambient infinite motion in the default view, and always respect reduced-motion.
- Report mobile should surface the first actionable card quickly without making stats unreadable. Compact stats are acceptable only while labels remain single-line and scannable around 390px; very narrow screens may trade density for readable single-column details.

## Product Boundary

- Signal Desk is the primary human surface. A user who only knows apps should be able to install, open, configure, log in, edit sources, set notifications, run dry-runs, and review output without typing commands.
- CLI and JSON contracts remain primary for agents, automation, and expert troubleshooting. They should not become the visible happy path for a human-first Desk workflow.
- The UI should show local/privacy state clearly without exposing bot tokens or raw Telegram text. Notification tokens may be saved only through the local OS credential store and must never be echoed back into UI state or SQLite.
- Empty state is valid product state and should still feel operational, not unfinished.

## Avoid

- Generic white SaaS cards.
- Decorative gradient blobs, glassmorphism, or one-hue palettes.
- Fake sample intelligence items in the real dashboard.
- Visible setup tutorials inside the primary app surface.
